# Verify API

Run after manually restarting the server in IntelliJ.
Reads the `## Test Plan` block generated by the agent (from backend-endpoint-convention Section 11)
and executes **only the steps listed there** — nothing more.

## Prerequisites

- Server must be running on `localhost:8080`.
- `jq` must be installed. macOS: `brew install jq`, Linux: `sudo apt install jq`, Windows: `winget install jqlang.jq`.
- `curl` must be available in the shell. On Windows PowerShell use `curl.exe` to avoid the `Invoke-WebRequest` alias.

## Trigger

User invokes: `@verify-api`

## Workflow

### Step 0 — Read the test plan from context

Find the most recent `## Test Plan` block in the conversation. Extract each step:
- Label (e.g. "Create", "Validation Error", "Confirm Soft Delete")
- Method + URL
- Body (if present)
- Expected status code
- Field assertions (if any)

If no test plan is found, stop and tell the user:
> No test plan found in context. Ask the agent to generate code first — it will output a test plan automatically per Section 11 of the backend convention.

If a step says "use id from step N", note the dependency — the id must be captured from that step's response and substituted into subsequent URLs.

### Step 1 — Health check

```bash
curl -s -o /dev/null -w "%{http_code}" http://localhost:8080/actuator/health
```

If not `200`, try the first GET endpoint from the test plan instead:
```bash
curl -s -o /dev/null -w "%{http_code}" {first_get_url_from_test_plan}
```

If still not reachable, stop and tell the user:
> Server is not running on port 8080. Please restart in IntelliJ first.

### Step 2 — Execute only the steps in the test plan

Run each step in order. For steps with a body:

```bash
RESPONSE=$(curl -s -X {METHOD} {URL} \
  -H "Content-Type: application/json" \
  -d '{body}')
echo "=== {LABEL} ===" && echo $RESPONSE | jq .
```

For steps without a body:

```bash
RESPONSE=$(curl -s -X {METHOD} {URL})
echo "=== {LABEL} ===" && echo $RESPONSE | jq .
```

**Handling id dependencies:**
If a step depends on an id from a previous step, capture it immediately after that step runs:

```bash
ID=$(echo $RESPONSE | jq '.data.id')
```

Then substitute `$ID` into the URL of dependent steps.

### Step 3 — Assert each step

For each step check:
- `.status` matches the expected status code from the test plan
- Any field assertions listed (e.g. `data.id is not null`, `data.name == "Test Room"`, `data == null`)

### Step 4 — Report results

Output a summary table of only the steps that were run:

```
| Step | Label                | Expected | Actual | Result    |
|------|----------------------|----------|--------|-----------|
| 1    | Create               | 201      | 201    | ✅ PASS   |
| 2    | Validation Error     | 400      | 400    | ✅ PASS   |
```

For any failing step:
- Print the full response body
- Suggest a fix referencing the relevant section of the backend-endpoint-convention rule

## Important Notes

- Only run steps that appear in the test plan. Do not add extra steps.
- Default port is `8080`. If the user specifies a different port, adjust all URLs accordingly.
- Steps that depend on each other (e.g. GET by id after POST) must always run in order — skip dependent steps if the step they depend on failed.
